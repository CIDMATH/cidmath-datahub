# 0030 — ICD-10-CM hierarchy: adjacency list + materialized path on the code table

## Status
Accepted — 2026-06-05

## Context
`codes.icd10cm` (the first table in the `codes` schema; built per the ICD-10-CM reference issue) currently stores a **flat** set of diagnosis codes — one row per `(icd10_code, edition_year)` with `description` and `is_billable`. That shape is sufficient for conformance (mapping a surveillance/clinical diagnosis code to a canonical, versioned reference) but it discards the **nested classification hierarchy** that ICD-10-CM is built around: 22 chapters, blocks (a.k.a. sections, e.g. `A00-A09`), three-character categories, and progressively more specific subcategories down to seven characters. Many real public-health questions need that tree: "roll daily case counts up to the ICD chapter", "every billable code under category `S72`", "what block does this code belong to". Without the hierarchy those become hand-maintained code lists.

The hierarchy is not in the CDC NCHS **order file** the current builder parses — that file is a flat, ordered list of codes with a header/billable flag and descriptions. The tree lives in the **tabular XML** (`icd10cm-tabular-<year>.xml`, inside `icd10cm-table and index-<year>.zip`), whose nesting is `chapter → section → diag → diag …`. A complication: the tabular XML stops at the listed `diag` level and expresses seventh-character codes (e.g. `S72.001A`) via `sevenChrDef` extension definitions rather than as explicit nodes, while the order file *does* enumerate every expanded billable code. So neither source alone is both complete (every code) and structural (the tree); the two must be reconciled. The decision is how to represent the tree in a standard, columnar Delta table that joins cleanly from fact tables and is queryable without recursive CTEs, and how to populate it from the two sources.

## Decision
Model the tree as an **adjacency list plus materialized path, denormalized with chapter/block labels, on the existing `codes.icd10cm` node table** — additive columns, no PK change. Each row gains:

```
parent_icd10_code  STRING          -- nearest ancestor code that exists this edition; NULL at category root
node_level         INT             -- depth in the adjacency tree (= length of ancestor_codes)
ancestor_codes     ARRAY<STRING>   -- root→parent path, e.g. ["S72","S72.0","S72.00","S72.001"] for S72.001A
chapter_code       STRING          -- "19"
chapter_name       STRING          -- "Injury, poisoning and certain other consequences of external causes"
block_code         STRING          -- section id range, "S70-S79"
block_name         STRING          -- "Injuries to the hip and thigh"
```

Population reads the **tree from the tabular XML**, which is its authoritative source. `parse_tabular_tree` walks `chapter → section → diag → diag …` recursively, recording each listed code's `parent_of` (the enclosing `diag`, or null for a three-character category) and the `category → chapter/block` map. `build_hierarchy` reads `parent_icd10_code`, `ancestor_codes`, and `node_level` straight from that nesting. The order file remains the spine for the code set, `is_billable`, and descriptions.

One unavoidable wrinkle drives the **fallback**: seventh-character codes (e.g. `S72.001A`, `S02.0XXA`) are *not* `diag` nodes in the XML — they are generated from a `sevenChrDef` extension — so the XML cannot supply their parent. For any code absent from `parent_of`, `resolve_ancestors` anchors it to the nearest *existing* prefix (the longest-existing-prefix rule: `S02.0XXA` → `S02.0`, since `S02.0XX`/`S02.0X` are not listed) and then reads that anchor's ancestors from the tree. The same prefix rule is the graceful degradation when the XML is skipped (`--hierarchy skip`): adjacency is computed from the order file's own code set and chapter/block are left null. Because every XML node is also an order-file code and ICD-10-CM nesting follows the code string, the XML tree and the prefix rule coincide for listed codes — so `find_adjacency_mismatches` records a WARN cross-check that flags any divergence (an XML node missing from the order file, or an unexpected nesting) rather than letting it pass silently. Logic stays pure and unit-tested in `cidmath_datahub.reference.icd10` (ADR 0011); the entrypoint adds the tabular-zip download alongside the order file. Everything is **edition-scoped** (the tree shifts between fiscal years) and recomputed on the existing full-refresh-per-edition write, consistent with the mid-year overlay already in place.

## Alternatives considered
- **Raw nested representation** (`ARRAY<STRUCT<children…>>` or the XML/JSON blob). Rejected: it defeats "standard Delta", joins poorly from fact tables, and BI tools handle it badly. The whole point is a relational, queryable tree.
- **Adjacency list only** (`parent_icd10_code`, no path). Rejected as the *sole* mechanism: correct, but subtree/rollup queries then require recursive CTEs. The materialized `ancestor_codes` array makes "everything under `S72`" a single `array_contains` with no recursion, and the rebuild-per-edition model makes the usual path-maintenance cost free.
- **Closure / bridge table** (`icd10_hierarchy_bridge(ancestor, descendant, depth)`, the `_bridge` Kimball form of ADR 0015). Deferred, not rejected: it makes arbitrary-depth set joins trivial but is largely redundant with `ancestor_codes` for the queries we expect, and ~74k codes × ~5 depth is only a few hundred thousand rows if we later want it. Add when join-based rollups are a demonstrated need.
- **Chapter/block from a static range table** instead of the XML (cf. `country_classifications`). Rejected as primary: chapter/block ranges are stable but new categories appear yearly, and the per-edition XML is authoritative and current. A `C4A`/`D3A`-style range expander was also avoided by reading category membership directly from the XML's section nesting rather than expanding `A00-A09` strings.
- **Deriving adjacency from the order file's code prefixes instead of the XML.** Considered and kept only as the *fallback*. The longest-existing-prefix rule provably reconstructs the same tree as the XML for listed codes (every XML node is an order-file code, and nesting follows the code string), and it needs no XML — attractive for resilience. But the XML is the authoritative structure, so it is the source of truth for listed codes; the prefix rule covers only what the XML cannot represent (seventh-character expansions) and the `--hierarchy skip` path. A WARN cross-check (`find_adjacency_mismatches`) asserts the two agree, turning the redundancy into validation.
- **Promote chapter/block to dim tables** (`codes.icd10_chapter`, `codes.icd10_block`). Deferred: cleaner normalization, but the denormalized columns are cheaper for analysts and the labels are short. Revisit if the labels need their own attributes.

## Consequences
- **The classification tree is preserved and queryable** in a standard columnar table: chapter/block rollups are a column or a `GROUP BY`; subtree selection is one `array_contains`; the full root-to-self path is `array_append(ancestor_codes, icd10_code)` — all without recursive SQL.
- **Purely additive.** PK is unchanged; the new columns land via Delta schema evolution on the per-edition write. Existing edition rows can be backfilled by re-running, and parent/path are recomputable from data already in the table.
- **A second source + format.** The builder now also downloads the ~18 MB table-and-index zip and parses XML. A reconciliation DQ check asserts the order-file and tabular-XML code/category sets agree; a code whose category is absent from the XML map resolves to a null chapter/block and is surfaced as a WARN rather than failing the build (e.g. a rare new mid-year category), keeping the pipeline resilient.
- **Seventh-character provenance is approximate by design.** Expanded codes (`…A`/`…XXA`) attach to their nearest *listed* ancestor, not a synthetic six-character stem; `node_level` reflects the adjacency tree, not raw code length. This is documented in `known_limitations`. Exact source ordering (the order file's `order_number`) and instructional notes (`excludes1/2`, `useAdditionalCode`, seventh-character definitions) remain out of scope — the latter belong in a future `codes.icd10_note` side table.
