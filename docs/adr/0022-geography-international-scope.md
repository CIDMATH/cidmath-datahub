# 0022 — Geography reference: international scope (countries, subdivisions, GADM)

## Status
Accepted — 2026-05-26. Temporal-model sub-decision (single-vintage snapshots, `full_refresh`) amended by [ADR 0024](0024-international-geography-vintaging.md) — 2026-05-29: the international tables are now vintaged like the US tables. The rest of this ADR stands.

## Context
ADR 0020 scoped `geography` to US units sourced from IPUMS NHGIS (state, county, ZCTA, tract, CBSA, HHS region, with the 2010↔2020 crosswalks in slice 2b). That scope is sufficient for US-centric surveillance work but blocks every modeling effort with an international dimension — multi-country pathogen surveillance (mpox, AMR, COVID variants), WHO-aligned reporting, global weather/climate joins (the next planned subject bundle), and any collaboration with non-US partners. WHO, IHR, and GBD pipelines all use **ISO 3166-1 alpha-3** as the canonical country key, so we cannot punt on country-level reference much longer without forcing every subject bundle to re-derive its own country lookup.

The international reference must answer the same questions the US reference does — *what is this place, what is it inside, and where is it on a map?* — but the data sources fragment by level. ISO 3166-1 and 3166-2 are pure code/name standards (no geometry). GADM publishes both codes and polygons for ADM0–ADM4 globally, but its hierarchical GID identifiers are GADM-specific (not the canonical surveillance keys). And below the first sub-national level there is no global standard at all — countries differ in whether they have districts, municipalities, or both, and at what depth.

We commit to **ISO 3166 codes as the canonical identifiers wherever they exist (country and first subdivision), and GADM as the single boundary source across all levels**, with GADM also providing both attributes and geometry for the deeper levels where ISO doesn't reach. GADM use is approved for the project (academic non-commercial; redistribution to consortium partners pending the same conversation as the IPUMS request and tracked the same way in `_ops.dataset_catalog`). Natural Earth was considered as an alternative geometry source but was rejected once GADM was available — running two boundary pipelines for the same levels would duplicate work and create reconciliation noise for no fidelity gain.

## Decision

### Source split: ISO codes, GADM polygons
**ISO 3166-1** (via the MIT-licensed `pycountry` package, which mirrors the official ISO 3166 maintenance agency list) supplies the canonical country attributes: alpha-3 PK, alpha-2 and numeric alternates, short and official names, and the active/historical (3166-3 predecessor) distinction. WHO region (AFRO/AMRO/EMRO/EURO/SEARO/WPRO) and UN M49 region/subregion are joined in from their respective published lists, both freely usable. **ISO 3166-2** supplies first-level subdivision codes, names, and subdivision-type labels (`state`, `province`, `region`, …), keyed by the standard `<country>-<sub>` form like `US-GA`. **GADM** supplies all geometry — ADM0 for countries, ADM1 for first subdivisions, ADM2/3/4 for deeper levels — plus the hierarchical attributes for the deeper levels where ISO doesn't go. ISO codes attach to GADM polygons via `ISO_A3` on ADM0 (clean 1:1 join) and `HASC_1` / `ISO_1` on ADM1 (clean for most countries; a few special cases — Kosovo, Western Sahara, Taiwan — handled by a small manual override table in code, not data).

### Tables and FK structure
Three new tables in the `geography` schema, all unsuffixed per ADR 0014/0015. Each is small enough to fit single-vintage (current ISO + current GADM release); update semantics `full_refresh` per ADR 0007.

- **`geography.country`** — PK `country_alpha3` (3-letter ISO, the canonical surveillance key). Columns: `country_alpha3`, `country_alpha2`, `country_numeric`, `country_name`, `country_official_name`, `who_region`, `un_region`, `un_subregion`, `is_un_member`, `is_sovereign`, `iso_3166_3_predecessor` (nullable, for historical codes like SU/CS), `centroid_geo_lon`/`centroid_geo_lat` (GADM ADM0 representative point), `ingested_at`, `source_file`. ~250 rows.
- **`geography.country_subdivision`** — PK `subdivision_code` (ISO 3166-2 like `US-GA`). Columns: `subdivision_code`, `country_alpha2` (FK → country, derived from the prefix), `country_alpha3` (denormalized for convenience), `subdivision_local_code` (the part after the dash), `subdivision_name`, `subdivision_type_label` (ISO's category string), `parent_subdivision_code` (nullable, for nested cases like UK's England → counties), `gadm_gid_1` (nullable, the matched GADM ADM1 id), `centroid_geo_lon`/`centroid_geo_lat`, `ingested_at`, `source_file`. ~5,000 rows.
- **`geography.subnational`** — PK `gadm_gid` (e.g., `USA.10.121_1`). Columns: `gadm_gid`, `gadm_level` (2/3/4), `subnational_name`, `subnational_type_label` (GADM's `TYPE_N`, e.g., `County`, `Município`, `Département`), `parent_gid` (FK either to another `subnational` row at level N-1 or to `country_subdivision` / `country` at the top of its chain), `country_alpha3` (denormalized FK), `subdivision_code` (nullable FK to `country_subdivision`), `centroid_geo_lon`/`centroid_geo_lat`, `ingested_at`, `source_file`. Volume varies by GADM release — tens of thousands of rows.

Boundary geometry extends the existing `geography.boundary` (ADR 0020) by adding new `geo_level` values: `country`, `country_subdivision`, `subnational_adm2`, `subnational_adm3`, `subnational_adm4`. Same WKB-and-generalized pattern as the US levels; the distinct values for the three GADM subnational depths let consumers filter to one level cleanly. `geography.boundary` stays Liquid-Clustered by `(geo_level, vintage)`.

**Storage guardrails:** ISO codes stored as `STRING`, never integer (numeric country code `004` is Afghanistan, not `4`). `gadm_gid` uses GADM's native format including the `_1` suffix it tacks on (e.g., `USA.10.121_1`) — strip it and joins break against the polygons. Centroids are always geographic (representative point); no population-weighted centroid pair is added here because no global population-weighted centroid source exists at usable consistency (US Centers of Population are US-only, and population denominators below country level live in the future `population` schema, not in `geography`).

**Relationship to `geography.us_state` and the US tables:** there is deliberate row-level overlap. `US-GA` exists in `country_subdivision` *and* in `geography.us_state` — the US-specific table keeps richer attributes (FIPS, USPS, HHS region) and remains the FK target for `us_county`/`us_tract`, while `country_subdivision` carries the globally-comparable ISO-2 view. Subject bundles join to whichever fits their grain. This duplication is a one-time cost to keep the US semantics tight; a `state_alpha2 → subdivision_code` mapping lives in the build code, not in data. The `us_` prefix on the US-specific tables (ADR 0006 refinement) keeps the distinction visible in the schema.

### Slices
Three slices, each shipping attributes + boundaries together for one administrative layer:

1. **Slice 3a — country.** `geography.country` from pycountry + WHO/UN region mappings, GADM ADM0 polygons into `geography.boundary` joined by `ISO_A3`. Smallest, validates the pycountry-+-GADM end-to-end shape.
2. **Slice 3b — first subdivision.** `geography.country_subdivision` from pycountry, GADM ADM1 polygons joined via `HASC_1`/`ISO_1` with a small in-code fixup map for the handful of countries where ISO and GADM disagree.
3. **Slice 3c — GADM subnational.** `geography.subnational` with GADM ADM2 / ADM3 / ADM4 attributes and boundaries; ADM2 (the county-equivalent for most countries) ships first within the slice, ADM3/4 follow per-country as coverage warrants. GADM is the only source for both attributes and geometry at these levels.

### Implementation
Pure logic — ISO-code parsing/validation, ISO↔GADM join fixups, GADM GID parent extraction, WHO/UN region resolution — lives in `src/cidmath_datahub/reference/geography_intl.py` (sibling to `geography.py`, unit-tested per ADR 0011). Bundle entrypoints `bundles/_reference/src/build_geography_country.py`, `build_geography_subdivision.py`, and `build_geography_subnational.py` stay thin; each gets its own job with a lighter or heavier environment as needed (3a needs only `pycountry` + `pyogrio`+`shapely`; 3b same; 3c adds the larger GADM downloads). All three reuse the `DQRecorder` infrastructure landed in the ops-tightening pass — uniqueness on the natural keys, FK integrity to parent levels, and `(country count == ISO 3166-1 list count ± historical)` cardinality checks.

### Access and licensing
Same reader-tier posture as the US `geography` schema (engineers + analysts) since the same downstream consumers want both. License obligations in `_ops.dataset_catalog`: pycountry (MIT, no DUA), GADM (academic non-commercial, citation required, redistribution permission pending), WHO region list (public), UN M49 (public). The GADM permission email is drafted in parallel with slices 3a/3b; if the response lands before 3c ships, redistribution metadata is updated then. If 3c needs to ship before GADM responds, the access posture stays internal-only (consortium-internal `ecdh-data-engineers` + `ecdh-analysts`, no external grants) and the metadata flags the pending state.

## Alternatives considered
- **Natural Earth boundaries for ADM0/ADM1 and GADM only for ADM2+.** Each slice would be source-pure. Rejected because GADM use is approved, so the *two-pipelines-for-same-levels* cost has no upside; GADM ADM0/ADM1 are also higher fidelity. Natural Earth remains a viable fallback if GADM redistribution permission is denied and external sharing of polygons becomes a requirement.
- **GADM attributes + boundaries for everything; skip ISO entirely.** Simplest single-source pipeline but loses the ISO 3166-1 alpha-3 / 3166-2 codes as canonical surveillance keys. WHO, IHR, and GBD pipelines key on ISO; using GADM's GID codes as the PK would force a code-translation step in every joining subject bundle. Rejected.
- **One unified `geography.subnational` table covering ISO 3166-2 and GADM levels.** Tempting for symmetry, but the keys are different (`US-GA` vs. `USA.10_1`) and the attribute sets are different (ISO has subdivision-type labels, GADM has admin-level depth). Two tables with their own PKs match the upstream shapes and avoid coalescing semantically different rows into the same surface.
- **Inline `geography.country.boundary_wkb` rather than `geography.boundary` extension.** Inconsistent with the US pattern (ADR 0020 chose companion-table geometry to keep attribute joins cheap), and a country polygon is large enough to make the inline cost real. Rejected for the same reasons that drove ADR 0020's split.
- **Defer everything until the first non-US data source forces the question.** Considered, but weather (the next subject bundle) is intrinsically global and would otherwise force this work on a critical-path schedule rather than now, when it can be done deliberately. Rejected.

## Consequences
- **Country and first-subdivision joins now work globally**, FK'd against canonical ISO keys that match WHO/IHR/GBD conventions. Future international surveillance subjects (mpox, AMR), weather, and any partner-shared modeling output gain a stable spatial backbone without each bundle re-inventing it.
- **GADM is a new external dependency** with its own license posture (academic use approved; redistribution pending email). The license obligation lives in `_ops.dataset_catalog` the same way IPUMS NHGIS's does; both must be revisited before any external-partner share goes live.
- **`geography.boundary` carries more rows** (countries ≈250, ADM1 ≈3,500, ADM2 ≈45,000, deeper levels add more). Liquid Clustering by `(geo_level, vintage)` already handles this — a US-only query prunes past the international rows.
- **Deliberate US/global overlap, made visible by naming.** `US-GA` exists in two tables (`geography.us_state` and `geography.country_subdivision`); we accept the duplication to keep US semantics tight rather than coercing US attributes into a global shape. The `us_` prefix (ADR 0006) makes the country scope of the US-specific tables obvious at the catalog level. Subject bundles pick the FK target that matches their grain.
- **Three new bundle jobs** under `_reference`, each with a `pycountry` + `pyogrio` + `shapely` (and 3c also a much larger GADM download) environment. Reuse the secret-scope and metadata-registration patterns from the existing geography jobs; no new bundle variables required.
- **No new ADRs needed for downstream subjects** — they FK to `geography.country` / `geography.country_subdivision` / `geography.subnational` the same way they FK to `geography.us_state` / `geography.us_county`.
